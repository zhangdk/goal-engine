# Goal Engine Agent 改进指南

**状态：** 建议稿  
**日期：** 2026-04-08  
**目的：** 基于真实外部目标观测结果，明确 Goal Engine 应该如何帮助 Agent 提升，而不是把场景答案硬编码进去。

---

## 1. 先给结论

这次观测暴露的核心问题，不是 Agent 完全不会做事，而是：

- 能开始探索外部目标
- 能识别部分阻塞
- 但还不能稳定把失败转成结构化学习，再进入下一轮

因此改进方向不应该是“把星巴克下单流程写死给 Agent”，而应该是：

**把学习协议、执行约束、恢复摘要和 token 控制做成稳定方法。**

一句话：

**不要固化答案，要固化学习方式。**

---

## 2. 这次观测到底说明了什么

以“星巴克热拿铁，中杯，少糖，静安寺附近门店自取”为例，当前系统已经证明了 3 件事：

1. Agent 能理解一个项目外、结果导向的真实目标
2. Agent 会主动探索外部路径，而不是只在项目内兜圈子
3. Agent 能识别一部分阻塞，例如环境访问限制、登录依赖、工具不可用

但还没有证明 3 件关键能力：

1. 失败能稳定写回 Goal Engine，形成新的 guidance
2. 下一轮会明确换路，而不是重复试错
3. 新 session 下能用最小恢复包继续逼近目标

所以当前瓶颈是：

**学习闭环不够稳定，执行骨架不够收敛，上下文控制还不够克制。**

---

## 3. 应该教 Agent 什么

### 3.1 教方法，不教答案

不应该教：

- “星巴克一定先点哪个按钮”
- “联合办公一定先搜哪个站”
- “支付失败一定去哪个页面重试”

应该教：

- 如何定义外部目标
- 如何判断一次尝试为什么失败
- 如何把失败压缩成可恢复的结构化摘要
- 如何在下一轮证明“这次和上次不同”
- 如何在上下文有限的情况下继续推进

也就是说，Agent 要学的是：

**外部目标执行方法论**

而不是：

**某个网站的固定操作答案**

### 3.2 要固化的是协议，不是内容

需要被固化的内容是：

- 失败后必须先归因，再继续行动
- 再次尝试前必须说明“本轮改变了什么”
- 对重复路径先做 retry check
- 新 session 恢复时优先读 recovery packet
- 每轮结束都生成最小恢复包

不需要被固化的内容是：

- 某个品牌、某个平台、某个网页的具体点击步骤

---

## 4. 建议补上的通用学习协议

每次外部目标执行失败后，Agent 必须先完成下面 4 步，再进入下一轮。

### 4.1 第一步：失败分类

失败必须先归入一个明确类别，而不是只写自然语言复盘。

推荐分类：

- `missing_user_input`
- `environment_blocked`
- `login_or_auth_required`
- `tool_unavailable`
- `strategy_mismatch`
- `validation_fail`
- `stuck_loop`

这样做的目的，是让系统知道：

- 这是用户缺信息
- 还是环境做不到
- 还是 Agent 自己策略错了

### 4.2 第二步：写出最小失败摘要

每次失败都要压成 5 个字段：

- `goal`
- `stage`
- `attempted_path`
- `failure_reason`
- `what_must_change_next`

禁止把整段长日志原样塞回上下文。

### 4.3 第三步：声明下一轮变化

下一轮开始前，Agent 必须显式回答：

- 这轮不再重复什么
- 这轮改走哪条路径
- 为什么这条新路径比上一轮更可能成功

如果回答不出这 3 个点，就说明没有发生真实学习。

### 4.4 第四步：重试前检查

下一轮执行前，必须做 retry check。

如果 planned action 仍与上一轮失败策略重叠，应该：

- 直接阻断
- 要求改写 planned action
- 或明确上报“当前没有新路径”

---

## 5. 外部目标的最小执行框架

外部目标不能完全自由发挥，否则 Agent 很容易：

- 一轮里开太多页面
- 一会儿搜官网，一会儿搜第三方，一会儿切 UI
- token 很快失控
- 失败原因也会变脏

更合理的方式，是给它一个最小骨架。

### 5.1 固定成状态机，而不是固定成网站脚本

推荐把外部任务统一拆成以下阶段：

1. `goal-clarification`
2. `channel-selection`
3. `candidate-validation`
4. `action-execution`
5. `pre-confirmation`
6. `handoff-or-block`

这不是告诉它“怎么点星巴克”，而是告诉它：

**做任何外部任务时，都先处在某个明确阶段。**

### 5.2 每轮只允许一个主路径

例如星巴克目标，一轮里只能选一个主路径：

- 官方网页
- 微信小程序
- 第三方平台

不能三条路并跑。

因为并跑会导致：

- token 暴涨
- 失败归因失真
- 难以证明哪条路有效

### 5.3 每轮只保留一个备选路径

主路径失败后，最多记录一个备选，不要一次列 5 个“可能可以试试”的方向。

恢复包里应该只出现：

- 当前主路径
- 当前备选路径
- 当前禁止路径

---

## 6. 什么时候是“环境问题”，什么时候是“Agent 问题”

这次观测里，`starbucks.com.cn` 被拦截、微信小程序需要登录、canvas 不可用，都不是同一种问题。

系统必须帮助 Agent 区分：

### 6.1 环境问题

例如：

- 当前网络访问被限制
- 浏览器能力缺失
- 工具依赖不存在

这类问题不应让 Agent无限自责重试。

正确做法：

- 记录为 `environment_blocked` 或 `tool_unavailable`
- 生成替代路径
- 若无替代路径，明确进入 handoff

### 6.2 Agent 问题

例如：

- 选错主路径
- 没澄清关键约束
- 明知不可行还重复试

这类问题才需要反思与策略更新。

正确做法：

- 记录为 `strategy_mismatch` 或 `stuck_loop`
- 生成 avoid strategy
- 下一轮强制说明变化

---

## 7. token 优化不是事后优化，而是主约束

这轮观测里，单轮 prompt token 已经上到很高量级。  
如果不压住，Goal Engine 会退化成“长期聊天历史管理器”。

### 7.1 只保留最小恢复包

恢复上下文建议固定成：

- 当前 goal
- 当前 stage
- 最近一次失败摘要
- 当前 guidance
- avoid strategies
- 下一步建议

不要把以下内容默认回灌：

- 全量 timeline
- 长段网页原文
- 完整多轮对话
- 所有候选列表

### 7.2 每轮都做上下文收缩

每轮结束时，Agent 应输出：

- `what happened`
- `what changed`
- `what next`

只保留这 3 段摘要进入下一轮。

### 7.3 外部探索要有预算

建议增加明确预算：

- 最多 3 个外部页面
- 最多 1 次主路径切换
- 最多 1 次长文内容抽取

超出预算时，必须先总结再继续。

## 8. 对 OpenClaw 的改进建议

如果要让 OpenClaw 自己找到执行所需的 skill，而不是总靠观测者提醒，需要补 3 类能力。

### 8.1 Skill 发现前置

当 Agent 接到“外部结果型目标”时，应该优先查找：

- 外部任务执行 skill
- `self-improvement` skill
- 长期目标恢复 skill
- 失败学习 / retry guard skill
- token 控制相关 skill
- 技能市场或 `find-skill` 能力中的可用专用 skill

这个步骤应该发生在真正开始大规模探索前。

### 8.2 Skill 触发条件要明确

例如满足以下任一条件时，优先加载外部目标 skill：

- 目标涉及外部网站、账号、预订、下单、预约
- 目标成功标准在项目仓库之外
- 目标存在登录、支付、环境阻塞风险

### 8.3 Skill 输出必须短

Skill 的价值不是增加 prompt 噪音，而是减少试错。

所以 skill 本身应该：

- 触发条件明确
- 步骤短
- 输出模板固定
- 避免长篇理念描述

### 8.4 长期记忆要有入口，但不要越权

OpenClaw Agent 在外部目标里应同时具备两层记忆：

- Goal Engine 状态记忆：goal、policy、retry、recovery
- Workspace 长期记忆：`MEMORY.md` 中的 durable lessons

其中：

- 失败细节优先写回 Goal Engine
- 反复出现的经验、用户偏好、执行边界，才进入长期记忆
- 不要用长 `MEMORY.md` 代替 Goal Engine 的控制层

---

## 9. 当前最值得做的产品改进顺序

### P0：必须先修

- 让 `record failed attempt` 更容易被 Agent 正确调用
- 修复 `recover current goal` 的稳定性
- 修复 `show goal status` 与 service 事实源不一致的问题

### P1：应该尽快补

- 把“失败分类 -> 摘要 -> 新策略 -> retry check”做成默认协议
- 给外部目标执行增加阶段模型
- 给恢复包增加更严格的 token 上限

### P2：可继续优化

- 让 OpenClaw 主动发现并选择合适 skill
- 给 UI 增加“本轮和上轮有什么不同”的对比视图
- 给观测者增加“当前学习闭环是否成立”的显式判断面板

---

## 10. 最终判断标准

如果 Goal Engine 真正帮 Agent 变强，最终应该看到的不是：

- 它更会说了
- 它更会复盘了
- 它调了更多工具

而应该是：

- 它在项目外目标上更容易收敛
- 它失败后能明确换路
- 它不再重复旧错误
- 它能在新 session 里继续推进
- 它没有靠无限历史上下文硬撑

这才叫：

**Agent 被教会了方法，而不是被塞进了答案。**
