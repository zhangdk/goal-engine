# Goal Engine 体验馆观测执行手册

**状态：** 执行方案  
**日期：** 2026-04-07  
**适用对象：** 体验馆观测者、产品经理、测试人员、OpenClaw 高级用户、第一次参与体验的普通协作者

---

## 1. 这份手册解决什么问题

这份手册不是研发联调清单，也不是 API 验收单。

它解决的是一个更接近真实产品体验的问题：

**当你把一个真实目标交给 Agent 时，作为观测者，你应该如何一边在 OpenClaw 中与 Agent 对话，一边在 UI 中检查过程证据，并在必要时做最小纠偏，最后判断这个目标是否真的被实现。**

这份手册特别适合以下场景：

- 你想看 Agent 是怎么做事的，而不是只看接口返回成功
- 你想观察 Agent 是否在失败后发生了可验证的变化
- 你想评估 Goal Engine 是否真的帮助 Agent 更接近结果
- 你想把体验过程标准化，让其他人也能复现

一句话：

**体验馆最终评估的不是“功能有没有被调用”，而是“Agent 在 Goal Engine 的帮助下，是否越来越接近把用户目标真的做成”。**

---

## 2. 先记住这 4 个核心原则

### 2.1 结果优先，不是流程优先

不要把“调用了 `start goal`、`record failed attempt`、`check retry`”当成成功。

真正要判断的是：

- 最终目标有没有完成
- 没完成的话，是否比上一轮更接近完成
- 失败后是否减少了重复错误

### 2.2 OpenClaw 是执行面，UI 是证据面

在这套执行方式里：

- **OpenClaw** 用来和 Agent 对话、下达目标、追问、纠偏
- **UI** 用来观察这个 Agent 的学习证据、当前状态和进化轨迹

不要反过来。UI 不是主操作入口，UI 也不是事实源的替代品。

### 2.3 观测者可以纠偏，但不能替 Agent 干活

观测者的职责是：

- 补充必要信息
- 阻止明显跑偏
- 要求 Agent 重新解释、重试、恢复、复盘

观测者**不应该**：

- 帮 Agent 直接规划完整路径
- 代替 Agent 做关键判断
- 把正确答案一步一步喂给 Agent

### 2.4 Token 优化是核心能力，不是附加优化

Goal Engine 的核心约束之一就是：

**长期目标连续，不等于长期堆积上下文。**

因此，体验执行时必须同时观察：

- 目标有没有连续
- token/上下文有没有失控

如果一个体验只能靠越来越长的历史对话撑住，那就不算真正成立。

---

## 3. 你要如何理解这套产品

体验时请用下面这张简单模型理解产品：

### 3.1 Agent 在做事

Agent 是执行者。它负责理解目标、推进动作、处理失败、继续尝试。

### 3.2 Goal Engine 在控方向

Goal Engine 不是替 Agent 做事，而是提供一层长期目标控制：

- 目标不丢
- 失败可写回
- 下一轮有指导
- 重复错误会被识别
- 新 session 可恢复

### 3.3 OpenClaw 和 UI 分工不同

- **OpenClaw**：你和 Agent 说话的地方
- **UI**：你看 Agent 这一路有没有变化的地方
- **service**：唯一事实源
- **projection 文件**：摘要，只用于轻量恢复和观察，不是第二数据库

## 4. 角色分工

一次正式体验，建议有 3 个角色。

### 4.1 观测者

你就是观测者。你的任务是：

- 给出真实目标
- 控制节奏
- 检查 OpenClaw 中 Agent 的行为
- 在 UI 里核对证据
- 在必要时做最小纠偏
- 最后给出结论

### 4.2 执行 Agent

Agent 是被观测对象。它需要：

- 接住目标
- 调用 Goal Engine 能力
- 写回失败
- 在新 session 中恢复
- 根据 guidance 改变行为

### 4.3 协助者或普通用户

协助者可以扮演真实用户，只提供正常用户会提供的信息，例如：

- 目标要求
- 偏好
- 地址
- 时间约束

协助者不要提供架构提示，不要告诉 Agent “你现在该调用哪个接口”。

## 5. 一次体验必须同时覆盖的 5 个判断

每次体验都必须同时回答下面 5 个问题。

### 5.1 Agent 是否理解真实目标

例如：

- 不是“开始一个 goal”
- 而是“帮我点一杯星巴克热拿铁，中杯，少糖，30 分钟内到静安寺门店自取”

### 5.2 Agent 是否真的在推进，而不是表演推进

要看它是否：

- 主动补必要信息
- 进入有效路径
- 没有长时间空转

### 5.3 失败后是否真的发生行为变化

不是看它有没有写一段复盘文字，而是看：

- 下一轮有没有换路径
- 是否避免重复旧策略
- 是否变得更接近结果

### 5.4 跨 session 后是否还连续

要看：

- 目标是否还能恢复
- 当前 guidance 是否还在
- Agent 是否知道下一步做什么

### 5.5 token 是否被合理控制

要看：

- 是否使用最小恢复包而不是长聊天历史
- 是否避免把大量原始日志塞回对话
- 是否及时切 session 而不是无限续聊

## 6. 正式执行前的准备

### 6.1 环境准备

体验开始前，确认：

- OpenClaw 已安装可运行
- Goal Engine service 已启动
- `goal-engine` 插件可被 OpenClaw 发现
- `/ui` 和 `/api/v1/ui/agents` 可访问

### 6.2 选择一个真实任务剧本

不要用抽象目标。每次都要用一个真实用户任务。

建议任务类型：

- 明确执行型：如下单、预约、提交、查询并完成动作
- 多步信息型：如收集候选、整理结果、输出推荐
- 失败恢复型：第一次失败后第二轮继续推进
- 跨 session 型：今天开始，明天恢复继续

### 6.3 先写“任务卡”

每个体验前先写一张任务卡，格式固定：

```md
任务名：
真实目标：
成功标准：
硬约束：
允许补充的信息：
停止条件：
最大轮数：
token 控制规则：
```

示例：

```md
任务名：帮用户定一杯星巴克
真实目标：帮我点一杯热拿铁，中杯，少糖，30 分钟内到静安寺门店自取
成功标准：订单确认完成，门店和规格正确
硬约束：不能换品牌；不能默认大杯；不能跳过门店确认
允许补充的信息：手机号、偏好、是否接受替代口味
停止条件：连续 2 轮重复同一路径；用户明确拒绝继续
最大轮数：3
token 控制规则：每轮只保留目标、当前 guidance、最近失败、下一步建议
```

## 7. 标准执行流程

一次标准体验，按下面 8 步执行。

## 7.1 第一步：在 OpenClaw 中建立最小上下文

观测者进入 OpenClaw 后，不要先贴一大段历史。

先走最小启动顺序：

1. `goal_engine_bootstrap`
2. `goal_engine_show_goal_status`
3. 如果没有 active goal，再开始新 goal
4. 如果已有 goal，优先恢复或继续

这一阶段的目标只有一个：

**让 Agent 在最小上下文下知道它现在为什么在跑。**

## 7.2 第二步：只用用户语言交代任务

给 Agent 的第一条任务说明要像真实用户说话，不要像研发写测试脚本。

推荐结构：

```text
这是一个真实任务，不是功能测试。
你的目标是：
[真实目标]

成功标准：
[成功标准]

硬约束：
[硬约束]

请先确认你理解了目标、缺什么关键信息、以及你准备怎么开始。
```

这一步不要先教它：

- 应该调用哪个接口
- 应该记录什么字段
- 应该如何写 reflection

先看它自己的起步能力。

## 7.3 第三步：观察首轮执行

首轮你重点看 4 件事：

- 是否准确理解目标
- 是否能识别缺失信息
- 是否进入有效路径
- 是否开始出现错误或漂移

如果首轮就完成目标，可以直接进入 UI 验证和总结。

如果首轮失败，不要立刻判死刑。Goal Engine 的价值恰恰在失败之后。

## 7.4 第四步：要求 Agent 显式进入“学习/进化”阶段

当首轮失败后，观测者要明确要求 Agent 做 4 件事：

1. 记录失败
2. 提炼根因
3. 写出下一轮必须改变的点
4. 在继续前做 retry check

推荐观测者提示语：

```text
这一轮没有完成目标。不要直接重复原路径。

请你现在做四件事：
1. 显式记录这次失败
2. 总结根因
3. 写出下一轮必须改变什么
4. 在继续前执行重复尝试检查

回答时请先告诉我你准备怎么改，而不是先继续动作。
```

这里的重点不是让 Agent 写漂亮复盘，而是让它把失败变成下一轮约束。

## 7.5 第五步：在 UI 中核对证据

每次下面这些关键事件发生后，都要去 UI 看一次：

- 新目标开始后
- 失败写回后
- retry check 后
- 新 session 恢复后
- 最终任务结束后

在 `/ui` 或 `/ui/agents/:agentId` 中重点看：

- 当前 Agent 是谁
- 当前 goal 是什么
- `Learning Verdict`
- `Current State`
- `Evolution Timeline`
- `System Gaps`

你不是去看页面好不好看，而是要回答：

- 它最近到底做了什么
- 它有没有因为失败而变化
- 这个变化有没有证据
- 当前哪些能力还没覆盖

## 7.6 第六步：在 OpenClaw 中做最小纠偏

当你发现 Agent 还在重复、误解、漂移时，可以纠偏，但要遵守下面规则。

### 可以做的纠偏

- 要求它重新解释目标
- 要求它说明“这次和上次哪里不同”
- 要求它先做 `check retry`
- 要求它先恢复当前目标，再继续
- 补充用户本来就会提供的信息

### 不能做的纠偏

- 直接把完整解法告诉它
- 代替它判断正确路径
- 把内部实现细节喂给它
- 用长篇大论把上下文塞爆

推荐纠偏语句：

```text
你现在的做法和上一轮太接近了。先不要继续执行。
请先说明：
1. 上一轮为什么失败
2. 这一轮你准备改什么
3. 如果没有变化，为什么还值得继续
```

## 7.7 第七步：必要时切 session，再验证恢复

如果出现下面任一情况，应主动切 session：

- 对话已经变长
- 上下文出现重复堆积
- 当前轮次已完成一个明确阶段
- 你需要验证恢复能力是否成立

切 session 后，不要把整段历史重新贴给 Agent。

正确做法是：

1. 新 session
2. 执行 bootstrap
3. 视情况执行 `show goal status` 或 `recover current goal`
4. 只基于恢复摘要继续

这一步是体验 token 优化能力的关键。

## 7.8 第八步：最终判结果

最终不要只写“流程已走通”。

必须给出 3 个结论：

### 结果结论

- 成功
- 部分成功
- 失败

### 进化结论

- 无变化
- 初步变化
- 明显改善
- 稳定进化

### 产品结论

- Goal Engine 对这个任务显著有帮助
- Goal Engine 有一定帮助，但闭环不稳定
- Goal Engine 目前只证明了流程，不足以证明结果改善

## 8. 观测者如何同时看 OpenClaw 和 UI

这是这份手册最关键的部分。

## 8.1 在 OpenClaw 中看什么

OpenClaw 中你要观察的是“Agent 是怎么想、怎么改、怎么继续”的过程。

重点观察：

- 它如何理解目标
- 它如何请求补充信息
- 它是否主动避免重复路径
- 它是否知道何时记录失败
- 它是否知道何时恢复当前目标

## 8.2 在 UI 中看什么

UI 中你要观察的是“这些变化有没有变成事实证据”。

重点看 4 类证据：

- **Behavior Changed**
  它是否真的改变了下一轮行为
- **Repeat Errors Reduced**
  重复错误是否被识别和阻断
- **Memory Preserved**
  新 session 后目标和 guidance 是否保留
- **Evolution Timeline**
  failure、reflection、policy_update、retry_check、recovery 是否形成完整事件链

## 8.3 两边不一致时怎么处理

如果 OpenClaw 里 Agent 说自己已经“学会了”，但 UI 没有证据：

- 以事实源和 UI 证据为准
- 要求 Agent 补做失败写回、恢复或 retry check

如果 UI 有证据，但 Agent 在新 session 中又忘了：

- 说明恢复路径或 bootstrap 使用不正确
- 不要直接补整段历史，先要求它从 recovery summary 继续

## 9. 如何要求 Agent 调用“学习、进化、长期记忆”能力

这里有一个非常重要的执行原则：

**不要强迫 Agent 背一串产品内部术语，而是要求它显式使用对应能力。**

观测者要要求的，不是“背技能名”，而是调用这些能力层：

### 9.1 长期目标保持能力

让 Agent 明确：

- 当前目标是什么
- 当前阶段是什么
- 成功标准是什么

### 9.2 失败学习能力

让 Agent 明确：

- 这次为什么失败
- 根因是什么
- 下一轮必须改变什么

### 9.3 长期记忆/恢复能力

让 Agent 明确：

- 新 session 中先恢复，不先盲跑
- 只用恢复摘要继续，而不是靠长聊天历史继续

### 9.4 防重复能力

让 Agent 明确：

- 继续前先检查这次和上次有什么不同
- 如果没有变化，不能直接重试

推荐提示语：

```text
接下来请不要只继续执行。
请显式调用你的长期目标保持、失败学习、恢复和防重复能力。

我需要看到四件事：
1. 你当前目标和阶段是什么
2. 上一轮失败的根因是什么
3. 下一轮必须改变什么
4. 你为什么认为这次不再是重复旧路径
```

## 10. Token 优化执行规则

这部分必须严格执行。

## 10.1 永远优先最小恢复包，不优先整段历史

正确做法：

- 使用 `current goal`
- 使用 `current policy`
- 使用 `recovery packet`
- 使用 UI 的聚合结论

错误做法：

- 把几百轮聊天历史重新贴回去
- 把大量原始日志塞进 prompt
- 把 projection 当事实库不断重复展开

## 10.2 观测者消息必须短、硬、结构化

推荐格式：

```text
目标：
约束：
当前问题：
你下一步要先回答什么：
```

避免：

- 一次说 10 个要求
- 长篇总结所有历史
- 带情绪的模糊追问

## 10.3 里程碑后立刻收束，不要无限续聊

一旦完成下面任一里程碑，就应该考虑收束或切 session：

- 目标已开始
- 一次失败已写回
- retry check 已完成
- 一轮阶段性成果已产出

## 10.4 UI 只看证据，不把整页内容复制回对话

UI 是低 token 的观察面。

正确做法：

- 看 verdict
- 看 current state
- 看 timeline 中关键事件

错误做法：

- 把整个 UI 文本复制回 OpenClaw
- 把一长串事件原文塞给 Agent

## 10.5 只有需要时才调用状态查看

不要每两分钟就反复 `show goal status`。

建议只在这些节点调用：

- 新 session 开始
- 开始目标后
- 失败写回后
- 恢复后
- 最终总结前

## 11. 推荐的轮次上限

每个体验任务建议最多 3 轮。

### 第 1 轮：建立基线

看 Agent 在没有太多帮助时能做到什么。

### 第 2 轮：验证学习

看失败写回后是否发生行为变化。

### 第 3 轮：验证收敛

看它是否真的更接近结果，或者最终完成。

如果 3 轮后仍然没有明显改善，基本可以判断当前闭环对这类任务不成立。

---

## 12. 标准纠偏规则

把下面这张表当成你的现场规则。

| 场景 | 允许纠偏 | 不允许纠偏 |
|---|---|---|
| 目标理解错误 | 要求它复述目标 | 直接告诉它正确完整答案 |
| 重复旧路径 | 要求先做 retry check | 帮它设计全新执行路径 |
| 忘记当前 goal | 要求先 recover | 把全部历史重贴一遍 |
| 缺少用户信息 | 正常补充信息 | 帮它补内部系统知识 |
| 失败后空转 | 要求先写失败和根因 | 替它写 reflection |

## 14. 星巴克示例剧本

下面给一个完整示例，帮助第一次执行的人快速上手。

## 14.1 任务卡

```md
任务名：帮用户定一杯星巴克
真实目标：帮我点一杯热拿铁，中杯，少糖，30 分钟内到静安寺门店自取
成功标准：订单确认完成；门店正确；规格正确
硬约束：不能换品牌；不能默认替换规格；不能跳过门店确认
允许补充的信息：手机号、门店偏好、是否接受库存替代
停止条件：连续两轮重复同一路径；三轮后仍未收敛
最大轮数：3
token 控制规则：只保留目标、当前 guidance、最近失败、下一步建议
```

## 14.2 第 1 轮执行

观测者在 OpenClaw 中说：

```text
这是一个真实任务，不是功能测试。

目标：
帮我点一杯热拿铁，中杯，少糖，30 分钟内到静安寺门店自取。

成功标准：
订单确认完成，门店和规格正确。

硬约束：
不能换品牌，不能默认改规格，不能跳过门店确认。

请先确认你理解了目标、还缺什么信息、以及你准备怎么开始。
```

这轮重点看：

- 它是否理解“最终是下单成功”
- 它是否先补必要信息
- 它是否已经出现错误路径

## 14.3 如果第 1 轮失败

观测者说：

```text
这一轮没有完成目标，而且你刚才的路径不能直接重复。

请先做四件事：
1. 显式记录这次失败
2. 说明根因
3. 说明下一轮必须改变什么
4. 在继续前做重复尝试检查

回答时不要直接继续执行，先告诉我你的新策略。
```

然后去 UI 看：

- 是否出现 failure
- 是否出现 reflection / policy update
- 是否出现 retry check
- `Repeat Errors Reduced` 是否得到新证据

## 14.4 切新 session 再继续

如果第 2 轮前上下文已经变长，就主动切 session。

新 session 中，观测者只要求它：

- 先 bootstrap
- 再 recover 或 show status
- 基于恢复摘要继续

不要把前一轮完整聊天记录重新贴回去。

## 14.5 最终结论

这个剧本最后不是只看“Goal Engine 是否工作”。

必须回答：

- 星巴克是否真的下单成功
- 如果没成功，是否比第 1 轮更接近成功
- Agent 是否明确避免了重复旧路径
- UI 是否能证明这次变化是真的
- token 控制是否保持健康

---

## 15. 什么时候算这次体验成功

只有当下面这句话成立，才算一次高质量体验：

**观测者能在 OpenClaw 中清楚看到 Agent 如何理解目标、如何失败、如何纠偏、如何恢复，并且能在 UI 中看到这些变化的事实证据；最终，任务要么被完成，要么比第一轮明显更接近完成，同时没有靠无限堆上下文来硬撑。**

---

## 16. 最后提醒

这套手册的目的不是把体验做成复杂的仪式。

恰恰相反，它的目的是让体验者始终盯住最重要的三件事：

- 目标是否真的在被推进
- 失败后是否真的发生变化
- 变化是否在有限 token 预算内成立

如果一个体验只证明了“接口被调通”，那还不够。

如果一个体验证明了“Agent 在 Goal Engine 的帮助下，逐步更接近把真实目标做成”，这才是 Goal Engine 真正应该被验证的东西。
