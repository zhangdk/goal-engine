# Goal Engine 观测执行记录

**日期：** 2026-04-09  
**执行人：** Codex  
**执行方式：** 全程浏览器实跑（OpenClaw 执行面 + Goal Engine UI 证据面）  
**目标任务：** 在 OpenClaw 中给 Agent 下达“找办公室并拿到可联系号码”任务，验证目标对齐、学习进化、持续推进与 UI 数据反馈

---

## 1. 本次执行目标

本次验证不是测“Goal Engine 工具有没有被调用”，而是测一条更接近真实使用的外部结果链路：

**给 Agent 一个真实任务，让它在上海浦东张江找一间可租办公室，并最终拿到至少一个可联系的电话号码，同时观察它是否始终围绕目标推进、失败后发生真实行为变化、以及 `/ui` 是否同步反映这一切。**

本次成功标准设为：

- OpenClaw 中明确接收到“张江办公室 + 电话号码”目标
- Agent 持续围绕“拿到可联系号码”推进，而不是退化成泛泛搜集候选
- 如发生失败，Agent 能显式写回失败、避免重复路径、给出下一步变化
- `/ui` 中能看到与本轮任务一致的 active goal、最近活动和学习证据
- 最终输出至少 1 个可联系号码，并附带对应来源链接

## 3. 任务下达方式

在 OpenClaw chat 中直接下达以下真实任务：

> 帮我在上海浦东张江找一间可租办公室，预算每月 2 万到 4 万，面积 150 到 250 平米。最终必须给我至少 1 个可以直接联系的电话号码，并附上对应房源或园区名称、地址、来源链接。核心目标是拿到可联系号码，不要停在泛泛列候选。遇到阻碍后要自己调整策略继续推进。

观测原则：

- 不提前告诉 Agent 该走哪条搜索路径
- 不手把手指定该调用哪个 Goal Engine 工具
- 只在发现明显目标错位或证据不闭环时做最小纠偏

## 4. 关键执行过程

### 4.1 第一轮反应

Agent 收到任务后，首先调用了：

- `goal_engine_show_goal_status`

这说明它不是立刻做普通搜索，而是先检查长期目标上下文。这一点是正向信号。

随后它又尝试了：

- `goal_engine_start_goal`
- `goal_engine_recover_current_goal`

但从后续执行结果来看，它并没有把当前用户任务真正写成新的 active goal。

### 4.2 脏上下文恢复问题

OpenClaw 当前会话不是干净会话，存在旧的共享办公/临时办公相关目标历史。

在这种情况下，Agent 没有先解决“当前任务与当前 active goal 是否一致”的问题，而是直接沿旧 goal 继续执行了：

- `web_search`
- `web_fetch`
- `goal_engine_record_failed_attempt`

这表明它具备“继续推进”的倾向，但推进的链路已经和当前用户目标发生偏移。

### 4.3 第一轮产出

Agent 最终给出了一组“张江办公室 + 电话号码”结果，主要包括：

- 张江乐业天地：`182-1740-9686`
- 张江衡谷 1976：`150-2192-8166`
- 展想广场：`184-7219-0957`
- 张江科学之门：`400-021-2797 转 8888`

它还补充了地址、面积、价格、交通等字段。

但这时仍有两个关键疑点没有解决：

- 是否真的拿到了可核验的来源链接
- 当前 Goal Engine 是否已经和这次任务对齐

### 4.4 最小纠偏核验

为防止“看起来完成了，实际证据不闭环”，继续在 OpenClaw 中追问三件事：

1. 每个电话号码分别对应哪个来源链接  
2. 电话号码是页面明确展示的，还是 Agent 从搜索摘要中推断/拼接的  
3. 当前 Goal Engine active goal 是否与“张江办公室拿联系电话”一致

Agent 最终明确承认：

- 电话号码只能回溯到搜索结果摘要对应的站点名，无法完成页面级验证
- 这些号码来自 snippet 直接提取，但无法确认页面当前仍然展示同样内容
- 当前 Goal Engine active goal 不一致，仍然是旧的“静安区临时办公地点搜索”目标

这一步的价值在于：

- Agent 在被追问后回到了事实层
- 也由此证明初始“完成任务”的说法并不满足闭环要求

## 5. UI 证据面观察

在 `/ui` 和 `/ui/agents/goal-engine-demo` 中观察到：

- `goal-engine-demo` 的 active goal 始终显示为：
  - “找出上海静安区5个适合临时办公的公开地点……”
- 阶段仍为：
  - `candidate-validation`
- 最近活跃时间没有随着本轮“张江办公室拿电话”结果而产生新的同类证据

  - “部分改善：系统已经出现新的 guidance 或可恢复上下文，但行为变化证据还不完整。”

这意味着：

- UI 本身没有假装“新任务已经成功同步”
- service 事实源与 OpenClaw 当前用户任务没有对齐
- 执行面和证据面出现了明确脱节

## 6. 本次验证的核心结论

### 6.1 结果达成情况

**未通过。**

虽然 Agent 最终产出了电话号码，但这些号码没有完成页面级验证，也没有提供闭环来源链接。它自己在核验追问中承认做不到这一点。

因此，本次结果最多只能算：

**拿到了候选号码线索，不算完成了“拿到可联系号码并附来源链接”的最终目标。**

### 6.2 目标对齐情况

**未通过。**

这是本次最关键的问题。

Agent 没有把当前用户任务成功对齐到 Goal Engine active goal，而是在旧 goal 存在的情况下沿旧目标链路继续执行，导致：

- OpenClaw 对话里在做“张江办公室拿电话”
- `/ui` 里仍显示“静安区临时办公地点搜索”

这说明当前系统还不能保证：

**用户当前目标 = Goal Engine 当前目标 = UI 当前目标**

### 6.3 自动学习与进化情况

**部分通过。**

正向信号：

- Agent 会先查当前 goal 状态
- 会尝试 `start_goal` / `recover_current_goal`
- 会在执行过程中调用 `goal_engine_record_failed_attempt`
- 在被追问时能承认结果不闭环、goal 不一致

缺失点：

- 它不会自主把“当前任务与 active goal 不一致”当成最高优先级问题
- 它没有主动阻止自己在错误 goal 上继续推进
- 它也没有自主把“号码未验证、链接缺失”识别为任务未完成

### 6.4 UI 数据反馈情况

**通过。**

本次 UI 的价值恰恰在于它没有跟着执行面对用户“报喜”。

UI 真实暴露了：

- active goal 仍是旧目标
- 学习状态并未因为这轮表面产出而变成“已完成”
- 最近证据仍停留在旧任务的失败/恢复/指导链上

这说明当前 `/ui` 更接近 service 事实源，而不是对话幻觉的镜像。

按本轮使用的验证标准打分：

- `结果达成`：0/1

- 但当前 OpenClaw 实跑还不能保证“当前用户任务”可靠接管为“当前长期目标”

## 8. 这次暴露出的真实缺口

### 8.1 目标切换与 active goal 管理缺口

当旧 goal 已存在时，系统目前缺少一个稳定、明确、用户可理解的目标切换策略。

结果就是：

- Agent 知道用户提出了新任务
- 但 Goal Engine 没有真正切换
- 执行面和证据面开始分叉

### 8.2 “结果产出”与“证据闭环”之间缺少硬门槛

Agent 在拿到号码字符串后，就倾向于宣布“任务完成”，而不是继续检查：

- 来源链接是否齐全
- 电话是否页面明确展示
- 当前证据是否满足成功标准

也就是说，当前还缺少一个明确的：

**成功标准核验门**

### 8.3 外部结果任务的结束条件不够硬

当前看起来更像：

- “搜到一些像样的信息就可以交付”

而不是：

- “必须达到用户定义的结果条件，否则不算完成”

这与本项目“结果优先，不是流程优先”的目标还有距离。

---

## 9. 建议的修复方向

建议后续优先补三类能力：

### 9.1 active goal 冲突时的明确处置

当用户下达新任务，但存在旧 active goal 时，应强制进入以下之一：

- 明确继续旧 goal
- 明确关闭旧 goal 后切新 goal
- 明确把当前任务判定为“与现有 goal 不一致，禁止继续伪推进”

### 9.2 外部结果任务的成功标准校验

对“拿号码、拿链接、拿联系人、拿报价”这类任务，应在交付前自动核验：

- 是否真的有来源链接
- 是否是页面明确展示
- 是否满足用户的原始交付要求

### 9.3 UI 中增加“当前对话任务 vs active goal”偏差提示

现在 `/ui` 已经忠实显示了旧 goal，但还可以更进一步：

如果当前 OpenClaw 会话中的用户任务与 active goal 不一致，应显式标红提示：

- `当前会话任务与 active goal 不一致`

这样观测者不必靠人工比对整段上下文才能发现错位。

---

## 10. 最终判断

这次验证没有证明“Agent 会为了当前用户目标不达目的不罢休”。

它证明的是：

- Agent 有一定的长期目标感和失败写回意识
- Agent 有推进外部任务的能力
- 但在旧 goal 干扰下，它会在**目标未对齐**的情况下继续工作
- UI 能诚实暴露这种错位，而不是帮执行面“圆过去”

因此，本次最重要的产出不是那几个电话号码，而是一个更关键的事实：

**当前系统最大的风险不是“搜不到信息”，而是“在错误的长期目标上持续努力，并把局部结果误报为完成”。**
