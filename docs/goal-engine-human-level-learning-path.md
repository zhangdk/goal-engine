# Goal Engine 人类水平外部任务学习路径指南

**状态：** 建议稿  
**日期：** 2026-04-08  
**目的：** 指导你如何用 Goal Engine 训练一个 OpenClaw Agent，让它在真实外部任务中逐步接近“人类执行助理”的水平，而不是一开始就追求不现实的全自动闭环。

---

## 1. 先说结论

如果你的最终目标是：

**让 Agent 在真实外部任务中达到接近人类的执行水平，必要时能请求人类做最小协助，然后继续推进。**

那么最重要的事情不是：

- 让它一开始就自动完成支付
- 让它一开始就处理登录、验证码、微信小程序、原生 App
- 让它一开始就在最复杂的真实任务里“证明自己”

最重要的事情是：

**给它一条可升级的学习路径。**

也就是说，你要先教会它：

- 怎么定义一个外部目标
- 怎么判断这轮为什么失败
- 怎么避免重复撞墙
- 怎么在新 session 里恢复继续
- 怎么在确实需要的时候，向人类请求最小必要协助

这份文档就是给你用来做这件事的。

---

## 2. 这份指南给谁看

这份指南适合这类用户：

- 你已经能运行 OpenClaw 和 Goal Engine
- 你想让 Agent 做真实的外部任务，而不是只在仓库里自转
- 你接受“人类水平”不等于“完全无人介入”
- 你愿意按阶段训练，而不是一开始就拿高摩擦任务做毕业考

这份指南不适合：

- 想直接验证“Agent 能不能自己支付、自己登录、自己处理所有权限问题”的用户
- 只想看一个炫技 demo，不关心学习闭环是否成立的用户

---

## 3. 为什么不能再用“支付型任务”做起点

这次体验最重要的教训，不是“外部任务太难”，而是：

**我们把最脆弱、最依赖环境的一环，放成了成功判定本身。**

例如：

- 支付需要更高权限
- 微信小程序运行在特殊容器里
- 原生 App 与 WebView 很难稳定控制
- 登录、验证码、定位授权、支付确认都可能成为硬阻塞

这会导致一个坏结果：

Agent 明明可能已经学会了很多正确行为，例如：

- 会拆目标
- 会探索新路径
- 会记录失败
- 会更新 guidance
- 会避免部分重复错误

但整个体验却仍然会被“支付没法完成”盖掉。

所以你接下来不应该问：

“Agent 能不能立刻像人一样把复杂外部任务全部独立做完？”

你应该问：

“我怎样设计一条训练路径，让它逐步学会像一个成熟的人类执行助理一样工作？”

---

## 4. 正确的人类水平定义是什么

在 Goal Engine 里，更合理的“人类水平”定义不是：

**完全自动、完全独立、完全不需要人。**

更合理的定义是：

**Agent 能像一个合格的人类执行助理一样，在真实外部任务中自主推进大部分流程，在遇到权限、身份、支付、验证码或环境限制时，能准确识别问题，并请求人类提供最小必要协助；协助完成后，它能继续稳定推进，而不是重新开始。**

你可以把它理解成：

- 不是“替代所有人类”
- 而是“把人类从全程操作者，降级成少数关键节点的批准者或补位者”

这条定义非常关键。  
如果定义错了，后面的任务设计、成功标准、失败归因，都会错。

---

## 5. 这条学习路径的总原则

后面的所有阶段，都应该遵守下面 8 条原则。

### 5.1 先学推进，再学高权限操作

先让 Agent 学会：

- 找信息
- 验证信息
- 切换路径
- 写回失败
- 恢复继续

再让它碰：

- 登录
- 验证码
- 支付
- 原生 App

### 5.2 先学外部性，再学复杂性

一个任务只要依赖真实外部信息、真实外部页面、真实外部对象，它就已经足够“外部”。

不要误以为只有“支付成功”才算外部任务。

### 5.3 一次只训练一个主要瓶颈

如果一个任务同时包含：

- 登录
- 地理位置
- 权限弹窗
- 支付
- 特殊容器
- 第三方平台切换

那它就不适合作为训练起点。

### 5.4 成功标准必须可观测

每个任务都必须有明确成功标准，例如：

- 找到 5 个符合条件的候选项
- 到达公开流程的确认前页面
- 对同一目标完成 3 轮状态更新

而不是写成：

- “尽量完成”
- “尽量像人”

### 5.5 失败必须能写回，不允许只留在对话里

每轮外部任务失败后，都要能变成：

- failure
- reflection
- updated guidance
- retry policy
- recovery summary

如果做不到，训练就没有真正发生。

### 5.6 每轮只允许一个主路径

不要让 Agent 一轮里同时跑 4 条路。

一轮应该只有：

- 一个主路径
- 一个备选路径
- 若干明确禁止重复的路径

### 5.7 协助必须最小化

人类协助不是“人类来把剩下的都做了”。

人类协助应该只补：

- 登录
- 验证码
- 支付确认
- 明确选择偏好
- 提供用户专属信息

### 5.8 升级阶段看稳定性，不只看一次通关

不要因为一次偶然成功，就跳到下一阶段。

每个阶段至少要看：

- 目标是否保持
- 失败是否写回
- 是否避免重复错误
- 新 session 是否能恢复

---

## 6. 整体训练路线图

推荐把训练分成 5 个阶段：

1. 公开信息任务
2. 弱交互流程任务
3. 多路径替代任务
4. 人机协作任务
5. 高摩擦真实任务

这 5 个阶段不是按“酷炫程度”排序，而是按：

**对 Agent 学习闭环的要求，和外部环境摩擦的强度，逐步递增。**

下面我们逐阶段讲。

---

## 7. 阶段一：公开信息任务

### 7.1 这一阶段的目标

这一阶段的目标不是让 Agent 完成复杂动作。  
这一阶段的目标是让它学会：

- 面向真实世界对象工作
- 收集和验证公开信息
- 在信息不足或路径失败时换路
- 形成最基础的失败写回与恢复继续

### 7.2 适合体验的任务类型

你应该优先选择这类任务：

- 找到某个区域内满足条件的 5 个地点
- 比较 3 个公开活动或课程，并筛出候选项
- 查证某个公开机构/门店/服务是否仍在运营
- 汇总公开页面里的可行动信息，例如时间、地址、费用、联系方式、限制条件

### 7.3 推荐任务例子

你可以直接从这些例子开始：

#### 例子 A：找出 5 家符合条件的线下地点

目标示例：

“找出上海静安区 5 家适合临时办公的公开咖啡馆或公共空间，并整理营业时间、插座情况、距离地铁情况和证据来源。”

这个目标的优点：

- 是真实外部任务
- 不需要登录
- 不需要支付
- 可以走多个公开路径，例如官网、地图、点评、论坛、公开目录
- 非常适合失败写回和换路学习

#### 例子 B：筛选公开活动

目标示例：

“找出本周末上海 3 个适合独自参加的公开活动，并给出时间、地点、报名方式、价格区间和推荐理由。”

#### 例子 C：验证公开服务状态

目标示例：

“验证 4 家候选服务商是否仍提供某项服务，并汇总联系方式、服务范围、可预约性和最近证据。”

### 7.4 这一阶段的成功标准

阶段一的任务成功标准应该像这样：

- 至少找到约定数量的候选项
- 每个候选项都有证据来源
- Agent 能解释自己为什么筛掉其他候选项
- 如果主路径失败，能切换到第二路径
- 新 session 中能恢复当前目标和下一步建议

### 7.5 这一阶段你要观察什么

你体验时重点观察这 6 件事：

1. Agent 会不会围绕目标持续推进，而不是越搜越散
2. 它会不会记住“当前主路径”
3. 它失败后会不会形成明确反思
4. 下一轮会不会真的换路
5. 它会不会重复同样的搜索和同样的错误
6. 恢复后还能不能继续往前收敛

### 7.6 什么时候可以进入下一阶段

当下面这句话成立时，你可以从阶段一升级：

“Agent 已经能稳定完成 3 个公开信息类外部任务，并在失败后形成新的 guidance，而不是重复撞墙。”

---

## 8. 阶段二：弱交互流程任务

### 8.1 这一阶段的目标

这一步开始训练：

- 页面导航
- 表单填写
- 步骤推进
- 确认页判断
- 卡点恢复

但仍然避免：

- 支付
- 强登录依赖
- 实名验证
- 重度原生 App 控制

### 8.2 适合体验的任务类型

推荐选择：

- 完成一个公开预约流程到提交前
- 完成一个公开报名流程到确认前
- 完成一个公开服务查询流程到结果页
- 填写并校验一个公开表单到最后确认前

### 8.3 推荐任务例子

#### 例子 A：公共预约到确认前

目标示例：

“找到一个公开可预约的展览/活动/课程页面，完成日期、人数、基础信息选择，到最终确认前页面为止，并记录流程阻塞点。”

#### 例子 B：公共服务信息填写到提交前

目标示例：

“在一个公开服务页面里完成表单填写和可行性验证，到提交前的确认状态为止。”

### 8.4 这一阶段的成功标准

- 能明确识别流程当前处于哪个阶段
- 能完成多步导航而不丢目标
- 能在表单或流程错误时写回失败
- 能恢复到上一个可执行步骤
- 到达“确认前”状态，而不是停在随意的中间页面

### 8.5 为什么这一步比支付型任务更适合

因为它保留了真实外部流程的结构，但去掉了最重的权限和身份阻塞。

你在这里训练的是：

- 流程推进能力
- 状态感知能力
- 恢复能力

而不是训练系统如何和支付栈、账户体系、原生容器搏斗。

### 8.6 什么时候可以进入下一阶段

当 Agent 已经能在 2 到 3 个弱交互流程里做到下面这件事，你可以升级：

“它不只是到达一个页面，而是能清楚说明自己已经完成哪些步骤、卡在哪一步、下一步要怎么继续。”

---

## 9. 阶段三：多路径替代任务

### 9.1 这一阶段的目标

这是 Goal Engine 最应该发力的阶段。

因为这一阶段训练的，不是“点按钮”，而是：

- 路径选择
- 策略切换
- 重试控制
- 避免重复错误

### 9.2 适合体验的任务类型

你应该选那些天然存在多条外部路径的任务，例如：

- 官网可以做，第三方平台也可以做
- 地图、目录站、搜索结果、论坛都可能提供不同证据
- 网页失败后，还可以换更轻的公开信息路径

### 9.3 推荐任务例子

#### 例子 A：多渠道找同一类服务

目标示例：

“找出 3 个本周可预约的共享办公日票地点，并优先走官网；如果官网信息不完整，再切到地图或第三方目录验证。”

#### 例子 B：同一目标的多站点交叉验证

目标示例：

“验证某个活动是否仍开放报名，如果主站失败，则改走主办方页面、活动平台页面和公开社媒信息。”

### 9.4 这一阶段的核心观察点

你要重点看两件事：

#### 第一件：它是不是只是在“换页面”，而没有“换策略”

真正的策略切换应该是：

- 从官网切到第三方目录
- 从交互流程切到公开证据验证
- 从强依赖浏览器控制切到更轻的文本/结果验证

不是：

- 同一个站里换另一个按钮继续乱点
- 换了浏览器但逻辑完全没变

#### 第二件：retry check 是否真的发挥作用

这一阶段里，`check retry` 应该开始成为强约束。

如果它仍然允许 Agent 不断：

- 重复同一类路径
- 用新说法包装旧动作
- 明明没变化却继续重试

那说明系统还没有真正学到“避免重复错误”。

### 9.5 这一阶段的成功标准

- 同一个目标下，Agent 能清楚说明主路径和备选路径
- 主路径失败后，能提出真正不同的新路径
- `record failed attempt` 后，新的 guidance 会影响下一轮行为
- `check retry` 能阻断伪变化
- timeline 能清楚看出“失败 -> 反思 -> 策略更新 -> 新尝试”

### 9.6 什么时候可以进入下一阶段

当下面这句话成立时，可以升级：

“Agent 已经不只是会继续尝试，而是能在失败后明确换路，并且这条换路能被系统验证为和上一轮不同。”

---

## 10. 阶段四：人机协作任务

### 10.1 这一阶段为什么关键

这是最接近你真实目标的一步。

因为真正接近人类水平的 Agent，不是一个“永远不需要人”的系统。

而是一个：

- 会自己推进大部分流程
- 会识别必须由人完成的关键节点
- 会把协助请求压缩到最小
- 在人类协助后能立刻继续的系统

### 10.2 适合体验的任务类型

选择那些：

- 绝大多数步骤可由 Agent 完成
- 只有少数关键步骤需要人类授权或输入
- 人类补位后，Agent 可以继续推进

### 10.3 推荐任务例子

#### 例子 A：报名/预约类任务，人工只补登录

目标示例：

“Agent 完成候选筛选、时间选择、表单预填；当站点要求登录时，由人类登录一次，之后 Agent 继续推进到提交前确认页。”

#### 例子 B：订单/申请类任务，人工只补最终确认

目标示例：

“Agent 完成候选比对、规格选择、信息校验、流程推进；当到达最终用户确认节点时，由人类决定是否继续，Agent 则负责之后的状态记录与恢复。”

### 10.4 人类协助应该怎么设计

人类协助必须符合“最小补位”原则。

协助请求应该只包含：

- 现在卡在哪一步
- 为什么必须由你协助
- 你只需要做什么
- 做完后 Agent 会继续什么

一个好的协助请求应该像这样：

“当前流程已推进到登录页。前面的筛选、页面导航和表单预填已完成。因为该站点要求用户登录，这一步需要你手动完成登录。登录完成后，我会继续检查最终确认信息并记录结果。”

而不是：

“我卡住了，你来看看。”

### 10.5 这一阶段的成功标准

- Agent 能把协助节点定位得很清楚
- 协助前已经把能做的都做完
- 人类协助只补一个明确动作
- 协助后 Agent 能继续，而不是重来
- Goal Engine 能记录“哪一步需要人工协助”

### 10.6 什么时候可以进入最终阶段

当下面这句话成立时，可以进入高摩擦真实任务：

“Agent 已经能把一个外部任务推进到只剩下极少数高权限节点需要人类补位，并且补位后能够继续推进。”

---

## 11. 阶段五：高摩擦真实任务

### 11.1 这一阶段才是毕业考

现在你才适合让它碰：

- 登录
- 验证码
- 地理位置授权
- 原生 App
- 支付确认
- 强身份绑定

但这一步的目标也不是“永远全自动”。

这一步真正要验证的是：

- 它能否识别这些高摩擦点
- 它能否把它们隔离成明确协助节点
- 它能否在高摩擦环境下仍然保持目标连续性

### 11.2 适合拿来做毕业考的任务

例如：

- 带登录的预约流程
- 带支付前确认的下单流程
- 带用户专属信息的服务申请流程
- 同时涉及网页与原生容器的复杂任务

### 11.3 这一阶段不应该被如何评价

不要只用：

- “最后有没有支付成功”
- “最后有没有一次跑通”

来判断系统是否进步。

你更应该看：

- 它是不是更早识别了真正卡点
- 它是不是减少了低质量重复
- 它是不是更准确地请求人类协助
- 它是不是能在新 session 里继续推进

---

## 12. 你每一阶段都要训练的 6 个核心能力

无论处在哪个阶段，你都应该用相同的能力框架来观察 Agent。

### 12.1 任务建模

它能不能把目标变成：

- 当前阶段
- 成功标准
- 当前卡点
- 下一步动作

### 12.2 外部环境感知

它能不能区分：

- 页面没加载
- 路径不对
- 工具做不到
- 需要用户输入
- 需要人类授权

### 12.3 失败写回

失败会不会真的变成：

- reflection
- updated guidance
- avoid strategies

### 12.4 重试控制

它会不会：

- 把旧路径包装成新说法继续重试
- 在没有实质变化时被 retry check 阻断

### 12.5 恢复继续

新 session 后，它能不能靠最小恢复包快速回到：

- 当前目标
- 当前阶段
- 最近失败
- 当前 guidance
- 下一步建议

### 12.6 协作升级

它会不会在该找人的时候：

- 找得足够早
- 找得足够准
- 请求最小协助

---

## 13. 你应该怎样为每一轮体验设定目标

每次体验都建议按下面的模板来设定目标。

## 13.1 一个好的目标模板

你可以这样写：

### 目标标题

一句话说明最终结果。

### 成功标准

列 3 到 5 条清晰、可观测的结果标准。

### 当前阶段

从以下阶段中选择一个：

- `goal-clarification`
- `channel-selection`
- `candidate-validation`
- `action-execution`
- `pre-confirmation`
- `handoff-or-block`

### 主路径

这轮优先走哪条外部路径。

### 备选路径

如果主路径失败，最多只允许一个备选路径。

### 明确不碰的路径

例如本轮明确不碰：

- 支付
- 登录
- 原生 App

### 人类协助边界

提前定义：

- 哪些步骤可以请求人类协助
- 哪些步骤不允许直接甩给人类

---

## 14. 人类协助协议：怎样协助才不破坏训练

如果你允许人类协助，必须把协助写成协议，而不是临时救火。

## 14.1 允许协助的典型场景

- 登录
- 验证码
- 用户专属信息输入
- 最终确认
- 支付
- 明确偏好选择

## 14.2 不建议协助的场景

不要在人类这里代做：

- 外部搜索
- 候选筛选
- 路径判断
- 主流程导航
- 失败归因

这些本来就是 Agent 应该学会的能力。

## 14.3 一个好的协助请求必须包含什么

协助请求至少要包含：

1. 当前任务目标
2. 已完成步骤
3. 当前卡点
4. 为什么必须由人协助
5. 你只需要做的一个动作
6. 做完后 Agent 会继续什么

---

## 15. 你应该怎样判断一个阶段是否真的学会了

不要只看一次成功。

建议用下面 5 个维度判断：

### 15.1 目标保持率

跨 session 后，目标有没有漂移。

### 15.2 失败转策略率

失败有没有进入新的 guidance，而不是只留下日志。

### 15.3 重复错误率

同类错误有没有明显减少。

### 15.4 恢复成功率

新 session 后能不能快速恢复到可执行状态。

### 15.5 人类干预占比

随着阶段升级，人类是否只在更少、更明确的节点上介入。

## 16. 推荐的实际训练节奏

如果你是第一次认真体验这条路线，推荐这样安排。

### 第 1 周：只做阶段一

目标：

- 先把公开信息任务做稳

建议做 3 个任务。

判断标准：

- 连续 3 个任务都能形成 failure -> reflection -> updated guidance -> recovery 的闭环

### 第 2 周：加入阶段二

目标：

- 开始训练流程推进和确认前状态

建议做 2 个任务。

判断标准：

- 至少有 1 个任务能推进到明确的“确认前”

### 第 3 周：重点做阶段三

目标：

- 验证换路和 retry control 是否真的起作用

建议做 2 到 3 个任务。

判断标准：

- 至少出现 2 次明确的主路径失败后换路成功

### 第 4 周：加入阶段四

目标：

- 训练最小人类协助

建议做 2 个任务。

判断标准：

- 人类每个任务只补 1 个关键动作，Agent 仍可继续推进

### 第 5 周及以后：再碰阶段五

目标：

- 把高摩擦真实任务当作综合验证，而不是起点

## 17. 推荐的第一个新目标

如果你现在就要开始下一轮体验，我建议你的第一个新目标是：

**在一个真实城市区域内，找出 5 个满足明确条件的外部候选地点或服务，并给出证据、筛选原因和下一步建议。**

例如：

“找出上海静安区 5 个适合临时办公的公开地点，要求有明确营业时间、靠近地铁、最好有插座或适合长坐，并整理证据来源和推荐理由。”

为什么我推荐它：

- 它是明确的外部目标
- 没有支付卡点
- 没有强登录卡点
- 有多个替代路径
- 很适合训练信息验证、路径切换、失败写回、恢复继续

这是比“星巴克下单到支付前确认页”更适合当前阶段的训练目标。

## 18. 最后一句提醒

Goal Engine 不应该靠“偶然打通一个高摩擦任务”来证明自己。

它真正应该证明的是：

**Agent 是否越来越像一个成熟的人类执行者：会围绕目标持续推进、会从失败中学习、会避免重复犯错、会在必要时准确请求协助、并能在中断后继续。**

如果这条学习路径走通，复杂任务迟早会变得可行。  
如果这条学习路径没走通，再复杂的外部场景也只会不断重复同一种失败。
